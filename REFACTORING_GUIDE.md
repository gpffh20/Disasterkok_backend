## 2. 서버 실행하기

### 2.1 데이터베이스 마이그레이션

마이그레이션은 "데이터베이스 테이블을 만드는 설계도"입니다.

```bash
# 1단계: 마이그레이션 파일 생성 (설계도 만들기)
python3 manage.py makemigrations --settings=config.settings.base

# 2단계: 마이그레이션 적용 (실제 테이블 만들기)
python3 manage.py migrate --settings=config.settings.base
```

> **성공하면 이런 메시지가 나옵니다:**
> ```
> Applying notification.0001_initial... OK
> Applying notification.0002_initial... OK
> ...
> ```

### 2.2 개발 서버 실행

```bash
python3 manage.py runserver --settings=config.settings.base
```

> **성공하면 이런 메시지가 나옵니다:**
> ```
> Starting development server at http://127.0.0.1:8000/
> Quit the server with CONTROL-C.
> ```


### 2.4 관리자 계정 만들기

관리자 페이지에 접속하려면 관리자 계정이 필요합니다:

```bash
python3 manage.py createsuperuser --settings=config.settings.base
```

물어보는 대로 입력하세요:
- Username: `admin`
- Email: `admin@test.com`
- Password: `admin123` (개발용이니 간단하게)

## 3. 데이터베이스 확인하기

### 3.1 SQLite 데이터베이스 파일

개발 환경에서는 `db.sqlite3` 파일이 데이터베이스입니다.

```bash
# 파일 위치 확인
ls -la db.sqlite3
```

### 3.2 Django Shell로 데이터 확인

```bash
python3 manage.py shell --settings=config.settings.base
```

Python 쉘이 열리면:

```python
# 모든 사용자 보기
from user.models import User
User.objects.all()

# 모든 게시글 보기
from post.models import Post
Post.objects.all()

# 모든 알림 보기
from notification.models import NotificationMessage
NotificationMessage.objects.all()

# 쉘 종료
exit()
```

### 3.3 DB Browser for SQLite (GUI 도구)

시각적으로 데이터베이스를 보고 싶다면:

1. [DB Browser for SQLite](https://sqlitebrowser.org/dl/) 다운로드
2. 프로그램 실행
3. "Open Database" 클릭
4. `db.sqlite3` 파일 선택
5. "Browse Data" 탭에서 테이블 선택

### 3.4 주요 테이블 설명

| 테이블 이름 | 설명 |
|------------|------|
| `user_user` | 사용자 정보 |
| `post` | 게시글 |
| `like` | 좋아요 기록 |
| `notification_setting` | 사용자별 알림 설정 |
| `notification_message` | 실제 알림 메시지 |
| `region` | 지역 정보 |

---

### 4.3 좋아요 → 알림 테스트

1. 사용자 A로 게시글 작성
2. 사용자 B 계정 생성
3. 사용자 B가 사용자 A의 게시글에 좋아요
4. 사용자 A의 알림 목록 확인 → 알림이 생성되어 있음!

## 5. 아키텍처 변경 사항

### 5.1 기존 구조의 문제점

```
┌─────────────────────────────────────────────────────────┐
│                    기존 구조 (Before)                     │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   User ──1:N──> Notification ──1:N──> Region           │
│                      │                                  │
│                      └── 문제: "알림 설정"만 있고        │
│                          "실제 알림 메시지"가 없음!      │
│                                                         │
│   • Notification 모델 = 사용자의 알림 설정 컨테이너     │
│   • 실제 알림(제목, 내용, 읽음여부)을 저장할 곳 없음    │
│   • 실시간 알림 기능 구현 불가                          │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**구체적인 문제들:**

1. **모델 네이밍 혼란**: `Notification`이 실제로는 "알림 설정"을 의미
2. **알림 데이터 부재**: 알림 제목, 내용, 읽음 여부를 저장할 수 없음
3. **확장성 부족**: 새로운 알림 타입 추가 불가
4. **실시간 기능 없음**: WebSocket 미지원

## 6. 기술 선택과 이유

### 6.1 Django Channels (WebSocket)

| 항목 | 내용 |
|------|------|
| **선택한 기술** | Django Channels 4.0 |
| **대안** | Socket.io (Node.js), Django-Websocket, Pusher |
| **선택 이유** | Django와 완벽 통합, 기존 인증 시스템 재사용 가능 |

```
왜 Django Channels인가?

┌─────────────────────────────────────────────────────────┐
│  Socket.io (Node.js)                                    │
│  ├── 장점: 성숙한 생태계, 많은 레퍼런스                 │
│  └── 단점: 별도 서버 필요, 인증 연동 복잡               │
├──────────── ─────────────────────────────────────────────┤
│  Django Channels ✓ (선택)                               │
│  ├── 장점: Django와 같은 코드베이스, JWT 재사용         │
│  └── 단점: Node.js 대비 커뮤니티 작음                   │
├─────────────────────────────────────────────────────────┤
│  Pusher/Firebase                                        │
│  ├── 장점: 인프라 관리 불필요                           │
│  └── 단점: 비용 발생, 외부 의존성                       │
└─────────────────────────────────────────────────────────┘
```

### 6.2 Redis (Channel Layer)

| 항목 | 내용 |
|------|------|
| **선택한 기술** | Redis + channels-redis |
| **대안** | In-Memory Channel Layer, RabbitMQ |
| **선택 이유** | 빠른 속도, 프로덕션 검증됨, 추후 캐싱에도 활용 가능 |

```
왜 Redis인가?

┌─────────────────────────────────────────────────────────┐
│  In-Memory Layer                                        │
│  ├── 장점: 설치 불필요, 간단                            │
│  └── 단점: 서버 재시작 시 연결 끊김, 다중 서버 불가     │
├─────────────────────────────────────────────────────────┤
│  Redis ✓ (선택)                                         │
│  ├── 장점: 빠름, 안정적, 다중 서버 지원                 │
│  └── 단점: 별도 설치 필요                               │
├─────────────────────────────────────────────────────────┤
│  RabbitMQ                                               │
│  ├── 장점: 메시지 영속성, 복잡한 라우팅                 │
│  └── 단점: 과도한 기능, 설정 복잡                       │
└─────────────────────────────────────────────────────────┘
```

### 6.3 Django Signals (알림 생성)

| 항목 | 내용 |
|------|------|
| **선택한 기술** | Django Signals (post_save) |
| **대안** | View에서 직접 생성, Celery Task |
| **선택 이유** | 코드 분리, 기존 View 수정 최소화 |

```
왜 Signal인가?

┌─────────────────────────────────────────────────────────┐
│  View에서 직접 생성                                     │
│  ├── 장점: 직관적, 디버깅 쉬움                          │
│  └── 단점: View 코드 비대화, 알림 로직 산재             │
├─────────────────────────────────────────────────────────┤
│  Django Signals ✓ (선택)                                │
│  ├── 장점: 관심사 분리, 기존 코드 변경 최소화           │
│  └── 단점: 암묵적 동작, 디버깅 어려울 수 있음           │
├─────────────────────────────────────────────────────────┤
│  Celery Task                                            │
│  ├── 장점: 비동기 처리, 재시도 가능                     │
│  └── 단점: 인프라 복잡도 증가 (Worker 필요)             │
└─────────────────────────────────────────────────────────┘
```

### 6.4 F() Expression (동시성 처리)

| 항목 | 내용 |
|------|------|
| **선택한 기술** | Django F() Expression |
| **대안** | select_for_update(), 낙관적 락 |
| **선택 이유** | 간단하면서도 Race Condition 해결 |

```python
# 기존 코드 (Race Condition 위험)
post = Post.objects.get(id=1)
post.like += 1  # 다른 요청이 동시에 실행되면 덮어씌워짐!
post.save()

# 개선된 코드 (F() Expression)
Post.objects.filter(id=1).update(like=F('like') + 1)
# SQL: UPDATE post SET like = like + 1 WHERE id = 1
# 데이터베이스 레벨에서 원자적으로 처리됨
```

## 7. Trade-offs (장단점)

### 7.1 모델 분리 (NotificationSetting / NotificationMessage)

```
┌─────────────────────────────────────────────────────────┐
│                      장점 (Pros)                         │
├─────────────────────────────────────────────────────────┤
│ ✅ 명확한 책임 분리                                      │
│    - NotificationSetting: 사용자 설정 관리              │
│    - NotificationMessage: 실제 알림 데이터              │
│                                                         │
│ ✅ 확장성                                                │
│    - 새로운 알림 타입 추가 용이                         │
│    - 알림별 메타데이터 저장 가능                        │
│                                                         │
│ ✅ 쿼리 최적화                                           │
│    - 알림 목록 조회 시 설정 테이블 JOIN 불필요          │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                      단점 (Cons)                         │
├─────────────────────────────────────────────────────────┤
│ ⚠️ 테이블 증가                                           │
│    - 1개 → 2개 테이블로 분리                            │
│    - 마이그레이션 필요                                  │
│                                                         │
│ ⚠️ 기존 코드 수정 필요                                   │
│    - FK 이름 변경 (notification → notification_setting) │
│    - 관련 View, Serializer 수정                         │
└─────────────────────────────────────────────────────────┘
```

### 7.2 WebSocket 사용

```
┌─────────────────────────────────────────────────────────┐
│                      장점 (Pros)                         │
├─────────────────────────────────────────────────────────┤
│ ✅ 실시간 알림                                           │
│    - 페이지 새로고침 없이 즉시 알림 수신                │
│    - 사용자 경험 향상                                   │
│                                                         │
│ ✅ 서버 부하 감소                                        │
│    - Polling 대비 요청 횟수 대폭 감소                   │
│    - 연결 유지로 핸드셰이크 오버헤드 제거              │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                      단점 (Cons)                         │
├─────────────────────────────────────────────────────────┤
│ ⚠️ 인프라 복잡도 증가                                    │
│    - Redis 서버 필요                                    │
│    - ASGI 서버 필요 (Daphne, Uvicorn)                   │
│                                                         │
│ ⚠️ 연결 관리                                             │
│    - 연결 끊김 처리 필요                                │
│    - 재연결 로직 클라이언트에서 구현 필요              │
│                                                         │
│ ⚠️ 스케일링                                              │
│    - 다중 서버 시 Redis 필수                            │
│    - Sticky Session 또는 Redis Pub/Sub 필요            │
└─────────────────────────────────────────────────────────┘
```

### 7.3 Signal 기반 알림 생성

```
┌─────────────────────────────────────────────────────────┐
│                      장점 (Pros)                         │
├─────────────────────────────────────────────────────────┤
│ ✅ 관심사 분리 (Separation of Concerns)                  │
│    - 좋아요 로직과 알림 로직 분리                       │
│    - PostLikeAPIView는 좋아요만 담당                    │
│                                                         │
│ ✅ 기존 코드 최소 변경                                   │
│    - View 코드 거의 수정 없음                           │
│    - 새로운 알림 트리거 추가 용이                       │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                      단점 (Cons)                         │
├─────────────────────────────────────────────────────────┤
│ ⚠️ 암묵적 동작                                           │
│    - 코드만 봐서는 알림 생성 파악 어려움                │
│    - "PostLike 저장하면 알림이 생긴다" 문서화 필요     │
│                                                         │
│ ⚠️ 디버깅 어려움                                         │
│    - Signal이 언제 호출되는지 추적 어려움               │
│    - 에러 발생 시 스택 트레이스 복잡                    │
│                                                         │
│ ⚠️ 테스트 복잡도                                         │
│    - Signal 비활성화/활성화 필요할 수 있음              │
└─────────────────────────────────────────────────────────┘
```

### 7.4 JWT 쿠키 저장

```
┌─────────────────────────────────────────────────────────┐
│                      장점 (Pros)                         │
├─────────────────────────────────────────────────────────┤

│    - httponly=True로 JavaScript 접근 차단              │
│                                                         │
│ ✅ 자동 전송                                             │
│    - 매 요청마다 자동으로 쿠키 포함                     │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                      단점 (Cons)                         │
├─────────────────────────────────────────────────────────┤
│ ⚠️ CSRF 취약점                                           │

│    - 다른 도메인에서 API 호출 시 쿠키 전송 제한        │
│    - 모바일 앱에서는 Header 방식이 더 적합             │
└─────────────────────────────────────────────────────────┘
```


**해결책:**
```bash

redis-cli ping  # PONG이 나오면 정상
```

**해결책:**
